Drishi Enterprise
Documentation

Drishi is a real-time AI interview assistant. It listens to live interview audio via microphone or Chrome Extension, transcribes questions with Whisper / Deepgram / Sarvam, retrieves optimised answers from a curated Q&A database (1,280+ entries), and streams Claude-generated responses — all in under 30ms (DB) or ~2–3s (LLM) — directly to the interviewer's browser via SSE. Covers Python, Java, JavaScript, SQL, Linux, DevOps, Telecom (SIP/IMS/SS7), AutoSys, Django, and production-support roles with per-user isolation and multi-candidate session support.

🔁 Two-Tier Caching

  • Runtime LRU cache (1000 entries) for identical questions
  • In-memory Jaccard token cache ordered by hit_count DESC
  • SQLite WAL mode (default) or PostgreSQL via DATABASE_URL

⚡ Prompt Caching

  • System prompt sent with cache_control: ephemeral
  • Reduces Claude token cost by ~80% on repeated context
  • Temperature 0.15 for deterministic, professional answers

🔐 Auth Model

  • Admin uses main session (Flask server-side)
  • Ext users get unique tokens via /ext-users
  • User Portal: /portal/<token> — no login required

Interview Pipeline

1️⃣

Capture & Transcribe

STT Layer

Audio captured by professional_audio.py (sounddevice → pyaudio → parec fallback chain). Local Whisper or cloud STT (Sarvam / Deepgram / AssemblyAI) transcribes the audio. A domain-specific initial_prompt with 200+ IT/telecom terms biases Whisper toward technical vocabulary. Post-transcription corrections fix common STT mishearings (e.g. "s i p" → "SIP").

2️⃣

Classify & Validate

AI Classification

question_validator.py rejects noise, filler, and non-IT speech (chemistry, food, garbled STT). Uses _IT_ADJACENT (400+ terms) covering all supported roles. Only valid IT questions reach the answer pipeline.

3️⃣

DB Lookup → Claude

Answer Generation

Jaccard similarity search (threshold 0.72) checks QA database first. On DB hit, answer is returned in ~1ms. On miss, Claude generates answer (12s timeout, temp=0.15, streamed).

🧠

Question Classifier

File: question_validator.py — Uses compiled regex + vocabulary sets, zero ML inference. Rejects non-IT noise (chemistry, food, politics), garbled STT output, and filler speech before it reaches the LLM. Plural forms (microservices, services, architectures, communicate) are handled correctly.

✅ Pass: IT Questions

  • Starter word (what/how/explain/...) + IT term
  • Python, Java, JS, SQL, Linux, DevOps, K8s
  • Telecom: SIP, IMS, SS7, Diameter, VoLTE
  • AutoSys, production support, ITSM
  • HR/behavioral: "tell me about yourself"
  • Plurals: "microservices", "services", "architectures"

⚙ _IT_ADJACENT Set

  • 400+ terms across all supported roles
  • Any overlap with question words → pass
  • Covers: Python keywords, Bash, Linux, K8s
  • SQL, Django, Flask, Java, JS, telecom, HR
  • DevOps, SRE, AutoSys, ITSM vocabulary
  • Plurals: services, containers, frameworks, etc.

❌ Reject: Non-IT Noise

  • Chemistry/science: arsenic, photosynthesis
  • Food/cooking: pepper, salt, butter
  • Politics/religion: election, church, mosque
  • Garbled STT: "pebble as crimson", "pepper reaction"
  • Filler: "uh yeah", "ok", short phrases
# Validation pipeline (question_validator.py) if _PURE_NOISE.issuperset(subject_words): return False, "chemistry/non-IT" if not has_tech and not (_words_set & _IT_ADJACENT): return False, "non_it_question" return True # passes to STT / DB / LLM pipeline

Follow-Up Detection

File: web/server.py → _is_followup_question() — 11/11 accuracy. Prevents wasting Claude API calls on clarifications.

✅ Detected as Follow-Up

  • "Can you elaborate on that?" (4 words, regex match)
  • "Please explain more" (3 words)
  • "Give me an example" (regex)
  • "Clarify that please" (regex)

❌ NOT a Follow-Up (handled as new question)

  • "Can you elaborate on ACID properties please give full detail" (10 words → NOT follow-up)
  • "Tell me more about microservices architecture" (6 words, but no regex match)

⚙ Logic: Two conditions (both must be true)

Word count < 9 AND matches _FOLLOWUP_RE patterns: can you, please, more, example, elaborate, clarify, expand on, again, repeat

Resume-Aware Answers

How It Works

When a user profile is loaded with Skills, Experience, or custom instructions, those details are injected into the Claude system prompt. The AI then tailors every answer to reference the candidate's actual background.

How to Configure

  • Go to /users → Edit profile
  • Add skills: Python, React, AWS, etc.
  • Add company/role: "Google — Senior SWE"
  • Add custom instructions for tone and focus
  • Select profile in main Interview page before starting
🏢

Company / Role Boost

Boosts Jaccard similarity scores for Q&A entries that match the current user's company or role, prioritising company-specific answers.

Score Boosts

  • Company match: +10% to Jaccard score
  • Role match: +8% to Jaccard score
  • Both match: +10% + 8% = +18% total
  • Applied before threshold check (0.72)

Configuration

  • Set Company and Role in user profile
  • Tag Q&A entries with company/role in QA Manager
  • Boost is automatic once profile is loaded
⚠

Duplicate Question Warning

How It Works

Every incoming question is checked against all questions asked in the current session using the same Jaccard similarity algorithm. If similarity ≥ 0.80, a warning badge is displayed to indicate the interviewer may be revisiting a topic.

Behaviour

  • Threshold: Jaccard ≥ 0.80 (stricter than DB match)
  • Warning shown in extension feed and user portal
  • Candidate can acknowledge and scroll to original answer
  • Session resets on STOP — next session starts fresh

Screenshot Problem Solver

Technical Flow

  • chrome.tabs.captureVisibleTab → base64 PNG
  • POST to /api/solve_from_image with {image: <base64>}
  • Server sends image to Claude Vision (claude-opus-4)
  • Claude extracts problem, writes solution with explanation
  • Response streamed back → rendered in extension popup

Best Practices

  • Ensure the problem text is fully visible before capturing
  • Works with LeetCode, HackerRank, Codility, CoderPad
  • Includes time/space complexity in output
  • May take 3–6s for complex problems (vision + generation)

Multi-User & Intro Isolation

Switch between candidate profiles instantly from the terminal bar USER dropdown or via POST /api/launch_config {"user_id_override": "N"}. Each user has their own introduction, resume context, and Q&A personalisation.

How to Switch Users

  • Terminal bar → USER dropdown → select name
  • Or /users → Activate button
  • Active user shown in terminal bar instantly
  • All new answers use the new user's context

Intro Isolation

  • "Tell me about yourself" always returns current user's intro
  • Intro is never cached in LRU — always fresh
  • On user switch: intro LRU entries evicted automatically
  • Session dedup check skipped for intro questions

Resume-Aware Context

  • Each user has skills, experience, role, company fields
  • LLM system prompt injected with active user's profile
  • Switching users updates LLM context immediately
  • Manage profiles at /users

QA Manager

Adding Q&A Pairs

  • Go to /qa-manager
  • Paste question + model answer
  • Tag with type (behavioral / technical / coding)
  • Optionally add company and role tags
  • Answer is tokenised and cached on save

Jaccard Matching System

  • Each Q&A stored with pre-tokenised question set
  • Lookup: Jaccard(incoming_tokens, stored_tokens) ≥ 0.72
  • In-memory cache sorted by hit_count DESC
  • Frequently matched entries served first (hot path)
